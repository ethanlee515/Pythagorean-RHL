\subsection{Reproduction and measured check}

With Rocq 9.0.1, \texttt{rocq-ssprove} development version, MathComp Analysis
1.16.0, and MathComp algebra tactics 1.2.7, the command
\begin{quote}
\footnotesize\ttfamily
make -f Makefile.rocq\\
theories/Security/\\
NoiseFloodingSecurity/Final.vo
\end{quote}
checks the theorem and its dependencies.  A forced rebuild in the provided
development environment took 472.54 seconds wall time (421.75 seconds user,
51.87 seconds system).  An incremental check of an up-to-date build took
0.25 seconds.  These are single observations, included for reviewer planning
rather than performance claims.

The repository \texttt{README.md} gives setup, build, theorem-inspection, and
directory-map instructions.  The manuscript build refreshes appendix excerpts
from the live sources, preventing displayed formal definitions from silently
diverging when both repositories are checked out together.

\subsection{Trusted computing base}
\label{sec:tcb}

The trusted base includes the Rocq kernel and standard library; MathComp,
MathComp Analysis, and their real-number/classical infrastructure; and
SSProve's program, heap, package, linking, and subdistribution semantics.  The
scheme, chart, correctness, parameter, and IND-CPA facts in
\Cref{tab:theorem-boundary} are theorem parameters and must be discharged by a
concrete instantiation.

The new KL, Gaussian, relational, compiler, and game-hop results contain no
local \texttt{Admitted}.  The installed dependency graph nevertheless reports
MathComp Analysis's
\path{realsum.__admitted__interchange_psum}: current SSProve distribution
semantics are built on these MathComp definitions and reach the upstream
placeholder~\cite{ssprove}.  This is the host framework's inherited assumption,
not an axiom introduced by our security development.

We independently discharge the mathematical obligation.  The artifact proves
the same statement as \texttt{interchange\_psum\_proved}, ending in
\texttt{Qed}; \texttt{Print Assumptions} for this replacement reports only the
usual extensionality and choice principles and not the admitted upstream
lemma.  The proof has also been submitted to MathComp Analysis for inclusion
in a future release.  Until an upstream release replaces the existing call
sites, \texttt{Print Assumptions} for the top-level theorem will continue to
display the placeholder's name.  We report that installed-library fact
explicitly, while distinguishing it from an outstanding local proof
obligation.  Claiming that the current compiled dependency graph is literally
axiom-free would still be inaccurate.

For an instantiated module \texttt{Secure}, reviewers can run
\texttt{Print Assumptions Secure.is\_secure}.  Besides library and classical
assumptions, expected output includes the concrete scheme/chart/correctness
interfaces, positivity of \(\gamma\), and the supplied IND-CPA theorem.  It
should not contain the program-logic rules, trace-compiler correctness, or
discrete-Gaussian KL identity as assumptions: those are proved in the
development.

\subsection{A cryptographer's audit path}

A reviewer need not inspect 27,986 lines to determine what is claimed.  The
recommended specification audit is:

\begin{enumerate}
\item inspect \path{Schemes/Indcpad.v} for the attack game, table discipline,
and query counter;
\item inspect \path{Constructions/NoiseFlooding.v} for the returned
distribution and width \((e^2+1)\gamma\);
\item inspect \path{Security/IndcpadSimulator.v} for the reduction's oracle
answers;
\item inspect \path{Schemes/ApproxFHE.v} for the chart and support-correctness
assumptions, which define the good-execution boundary for the standard
up-to-bad extension;
\item inspect the definitions of \texttt{security\_loss},
\texttt{security\_bound}, and \texttt{is\_secure} in
\path{NoiseFloodingSecurity/Prelude.v} and \path{Final.v}.
\end{enumerate}

The formal excerpts in \Cref{app:formal-interface-excerpts} follow this path.
The remaining proof scripts establish that those definitions are connected by
the argument in \Cref{sec:verified-reduction}.  As in SSProve's own formal
security bounds, probabilistic-polynomial-time preservation is a conventional
metatheoretic premise rather than an internal semantic judgment~\cite{ssprove}.
The audit is direct here: the reduction runs the adversary once, forwards its
oracle calls, and adds table operations and discrete-Gaussian sampling.  The
artifact therefore exposes all reduction code needed for this audit; it does
not attempt to add automatic complexity analysis to SSProve.
