\subsubsection{On Locally Euclidean Metric Spaces}
\label{sec:isometry}

In lattice cryptography, one often considers polynomials and integers modulo some ideal generated by some polynomial $x^n+1$ and some number $q\in\bbN$.
Instead, we generalize the analysis to any locally-$\bbZ^n$ metric space.

We recall the textbook definition of a metric space in \Cref{def:metric-space}.
We now generalize the \emph{approximate correctness} notion for FHEs to metric spaces:
\begin{definition}
    Let $\varepsilon>0$ and $\delta>0$ be fixed parameters.
    Let $(X, d)$ be a metric space.
    A HE scheme $S$ is $(\delta, \varepsilon)$-\emph{approximately correct} if for all valid keys $(pk, evk, sk)\in\supp(keygen)$ and valid functions $f:X^*\rightarrow X$ and input $m\in X^*$,
    $$\Pr[d(f(m), (Dec\circ Eval_f\circ Enc)(m))\geq \delta]\leq \varepsilon$$
    where the probability is taken over randomness of the encryption, evaluation, and decryption algorithms.
\end{definition}

In general FHEs such as CKKS, we have $\varepsilon>0$ and $\delta>0$ for FHEs.
For simplicity, we verify the $\varepsilon=0$ case.
Finally, we need to connect discrete Gaussian distributions to this metric space setting.
As discrete Gaussian distributions are well-defined on $\bbZ^n$, we simply require a local isometry between $\bbZ^n$ with our metric space as defined by \Cref{def:local-isometry}.
We remark that the isometry indeed needs to be \emph{local}, as there is no global isometry between $\bbZ$ and the cyclic group $\bbZ/q\bbZ$ of lattice cryptography.

\subsubsection{Treatment of Query-Bounded Attackers}
\label{sec:query-bound}

Attackers against cryptographic constructions are often modeled as oracle algorithms.
Intuitively, an attacker against an encryption scheme aims to decrypt some target ciphertexts without knowing the secret decryption key;
such an adversary would be given an encryption oracle that generates target ciphertexts.

In the noise flooding analysis, as we will see in \Cref{sec:games}, the adversary is allowed to make at most $q\in\bbN$ queries to an oracle.
An earlier version of the development assumed the attacker was factorized into $q$ subroutines, one corresponding to each oracle query.
The current trace-compilation formalism described in the previous section is intended to remove this as a semantic assumption for concrete SSProve adversaries: one can compile the first $q$ calls to a selected operation identifier and reason about the resulting code.
In the Coq development, this is paired with an explicit decryption-query counter in both the real and simulated oracles.
Calls beyond the bound assert-fail on both sides, so compiling the first $q$ calls to the decryption identifier is enough to cover every non-failing decryption response.

There remains a smaller limitation of our current Pythagorean judgment:

There is no consistent way of writing down a Pythagorean judgment relating these two programs, as our judgment is parameterized by a fixed number of sampling operations,
and does not capture the case where the number of samples itself is also random.

On pen-and-paper proofs, this is handled implicitly by a generalization of what the Easycrypt community refers to as \emph{reflection} \cite{reflection}.
That is, all code snippets and procedures are synonymous and interchangable with their respective output distributions.
For instance, Pythagorean probability preservation (\Cref{def:pythagorean-probability-preservation}) is applied directly to the distribution of the entire transcript of the protocol as if the adversary is already a distribution over its oracle query inputs and its eventual response to the challenger.

In existing literature,
this issue is typically handled using \emph{call invariants} and \emph{adversarial call rules} in the union bound context.
For call invariants, one intuitively write down a sub-coupling that holds up to some \emph{bad event};
for example, given two distributions $D_x$ and $D_{x'}$ such that $\Delta(D_x, D_{x'})<\varepsilon$,
one can write down a coupling $D_{x, x'}$ so that $\Pr[x=x']=1-\varepsilon$.
In our KL divergence settings, however, it is unclear how to generalize this coupling machinery.
As for adversarial call rules,
thematically they do not fit in the foundational setting of SSProve.
In Easycrypt, adversarial call rules are necessary as adversaries are opaque objects that one cannot interact with except through adversarial call rules.
As the adversaries are concrete programs in SSProve, to the best of our knowledge, there has not been any attempts in implementing any Easycrypt-style adversarial call rules in SSProve.
If we were to write down an adversarial call rule for our setting, this would only be pushing these factorization challenges one layer down,
where now the soundness of the rule needs to be verified.

For the general oracle setting,
our current strategy is precisely to construct this factorization as code and verify its correctness.
The trace machinery steps through the SSProve syntax to read out the first $q$ selected oracle queries, records enough of the surrounding computation to resume the program, and proves that linking the compiled program with the same package preserves the original semantics.

\subsubsection{Security Games}
\label{sec:games}

Now we state the main lemma that we want to formally verify.
We aim to show that the two experiments (or \emph{cryptographic games}) given in \Cref{alg:indcpa-games} are indistinguishable.
That is, as the FHE scheme is approximately correct, the adversary cannot distinguish between any homomorphically evaluated computation and unencrypted computation.
This is relevant in the FHE setting, as the noises and errors could leak information otherwise, which was indeed the principle behind the LM attack.
More formally, we aim to prove the following theorem:
\begin{theorem}
    \label{thm:noise-flooding-secure}
    Let $\delta=O(\sqrt{q})$.
    \Ethan{TODO Cannot have asymtotics here. Need to trudge through concrete parameters and expressions.}
    For all possible adversaries $\cA$ we have
    $$\TVD(\llbracket G_L\rrbracket, \llbracket G_R\rrbracket)<\delta$$
\end{theorem}
Notice that in \Cref{alg:indcpa-games}, the attacker is indeed already factored into $q$ subroutines as explained previously in \Cref{sec:query-bound}.

\begin{algorithm}
\caption{The Hybrids}
\label{alg:indcpa-games}
Ingredients:
\begin{itemize}
    \item Homomorphic encryption scheme $S=(kg, enc, eval, dec)$ with key types $PK, SK, EVK$ and message type $M$ and ciphertext type $C$.
    \item Query upper bound $q\in \bbN$
\end{itemize}
Global memory:
\begin{itemize}
    \item Keys $pk, evk, sk$
    \item Message ID: $b\in\set{0, 1}$
    \item Adversary query history: $D\in (M \times M \times C)^*$
    \item Number of decryption queries so far $d$
\end{itemize}
Adversary restrictions:
\begin{itemize}
    \item The attacker main routine $\cA\Call{.run}{}$ cannot read $b$, $D$, or $sk$.
    $\cA$ can access $pk$ and $evk$, as well as any other memory locations.
    \item The attacker can call the subroutines OEnc, OEval given in \Cref{alg:indcpa-subroutines} for any number of times.
    \item The adversary makes at most $q$ queries to the $ODec$ decryption oracle, enforced in the formal games by a decryption-query counter.
    \begin{itemize}
        \item
        In the game $G_L$, ODec invokes ODecL.
        \item
        In the game $G_R$, ODec invokes ODecR.
    \end{itemize}
\end{itemize}
Games $G_L$ and $G_R$:
\begin{algorithmic}
    \Procedure{Game}{}
    \State $(pk, evk, sk) \gets kg$
    \State $b\gets \set{0, 1}$
    \For{$t\in\set{1, \ldots, q}$}
    \State $i\gets \cA^{OEnc, OEval}\Call{.DecQuery}{t}$
    \State $y_i\gets ODec(i)$
    \EndFor
    \State $b^*\gets\cA^{OEnc, OEval}\Call{.guess}{}$
    \State $win\gets b^* = b$
    \State \Return $win$
    \EndProcedure
\end{algorithmic}
\end{algorithm}

\begin{algorithm}
\caption{Oracles}
\label{alg:indcpa-subroutines}
Same ingredient and shares global memory with the top-level \Cref{alg:indcpa-games}.

Subroutines:
\begin{algorithmic}
    \Procedure{OEnc}{$m_0, m_1$}
        \State $c\gets enc(pk, m_b)$
        \State $D\gets D \doubleplus (m_0, m_1, c)$
        \State\Return $c$
    \EndProcedure
    \State
    \Procedure{OEval}{$f, \cI$}
        \State $m_0'\gets f(m_{\cI, 0})$
        \State $m_1'\gets f(m_{\cI, 1})$
        \State $c'\gets Eval(f, c_{\cI})$
        \State $D\gets D \doubleplus (m_0', m_1', c')$
        \State \Return $c'$
    \EndProcedure
    \State
    \Procedure{ODecL}{$i$}
        \State $y\gets Dec(D_i(c))$
        \State $y'\gets\DG{y}{\sigma}$
        \State \Return $y'$
    \EndProcedure
    \State
    \Procedure{ODecR}{$i$}
        \State $y\gets D_i(m_b)$
        \State $y'\gets\DG{y}{\sigma}$
    \State\Return $y'$
    \EndProcedure
\end{algorithmic}
\end{algorithm}

\subsection{Proof Overview}

We now present a top-down sketch for the main steps in the security proof of \Cref{thm:noise-flooding-secure}.
Our goal then is to prove
\begin{equation}
    \vDash\AEJ\set{\top}G_L\sim_{\delta} G_R \set{w_L = w_R}
\end{equation}
for all possible adversaries $\cA$.
Using standard sequence rules, it suffices to analyze the loop containing the $q$ queries:
\newcommand{\tableempty}{table\_empty}
\begin{align*}
    \vDash&\AEJ\set{m_L=m_R\wedge keys\_valid\wedge\tableempty} \\
    &loop_L\sim_{\delta} loop_R \\
    &\set{m_L=m_R\wedge keys\_valid}
\end{align*}
where $keys\_valid$ denotes that the keys $(pk, evk, sk)$ in the memory satisfies $(pk, evk, sk)\in \supp (keygen)$ and $\tableempty$ denotes that the query table $D$ is empty.
Using our \LabTirName{Micciancio-Walter} rule then, it suffices to show that
\begin{equation}
    \vDash\Pyth\set{m_L = m_R \wedge keys\_valid}loop_L\approx_{\mathcal{E}} loop_R \set{keys\_valid}
\end{equation}
for some vector $\mathcal{E}$ of size $q$.
This naturally reduces to showing the following for each iteration $t$ in the loop body:
\begin{equation}
    \label{eq:loop-body-triple}
    \vDash\Pyth\set{m_L = m_R \wedge \Phi}iteration(t)_L\approx_{\set{\varepsilon}} iteration(t)_R \set{\Phi}
\end{equation}
where the loop invariant $\Phi$ contains validity of the keys and the query history table,
and $\varepsilon$ is simply given by the standard KL-divergence between two discrete Gaussians from \Cref{thm:discrete-gaussian-kl}.

To prove \Cref{eq:loop-body-triple}, 
we unfold the loop body and write down the properties satisfied by each of the three instructures inside.
First we have that
\begin{equation}
    \vDash\AEJ\set{\Phi\wedge m_L=m_R} A.query\sim_{0} A.query\set{\Phi\wedge i_L=i_R\wedge m_L=m_R}
\end{equation}
by symmetry, with the same invariant $\Phi$ as before on validity of keys and tables.
We have also that
\begin{align*}
    \vDash&\AEJ\set{\Phi\wedge i_L=i_R\wedge m_L=m_R}\\
    &\qquad y_L\gets Dec(D_i(c))\sim_0 y_R\gets D_i(m_b)\\
    &\set{\Phi\wedge\abs{y_L-y_R}<\delta_{corr}\wedge m_L=m_R}
\end{align*}
where $\delta_{corr}$ is the approximate correctness error of the underlying FHE.
Finally, using the \LabTirName{KL-DG} rule, we have
\begin{align*}
    \vDash&\Pyth\set{\Phi\wedge m_L=m_R \wedge\abs{y_L-y_R} < \delta_{corr}}\\
    &y'\gets\DG{y_L}{\sigma}\approx_{\set{\varepsilon}} y'\gets\DG{y_R}{\sigma}\set{\Phi}
\end{align*}
By combining the loop body through \LabTirName{AE-Seq} and \LabTirName{AE-Pyth-Seq} then, we arrive at \Cref{eq:loop-body-triple}.
